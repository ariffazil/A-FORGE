## 1. Authorization

### Who Can Query GEOX

Every `GeoRequest` must include a `requester_id`. This ID is verified against the arifOS authority chain per **F11 (Authority)**:

1. GEOX receives `GeoRequest` at `000 INIT`
2. `requester_id` is checked against allowed requester list (from `config_geox.yaml` or arifOS M2 Governance Gate)
3. For `risk_tolerance = "high"` or `risk_tolerance = "critical"`: requester must hold elevated clearance level
4. Unrecognized requester ID → request rejected, 888 HOLD not issued (no action to hold), error returned

### Clearance Levels

| Level | Who | Allowed risk_tolerance |
|-------|-----|------------------------|
| L0 | Public / demo | low only |
| L1 | Geoscientist | low, medium |
| L2 | Chief Geoscientist | low, medium, high |
| L3 | 888_JUDGE (Arif) | all including critical |

## 2. Risk Level Definitions

### LOW — Regional Screening

**Definition**: Basin-scale play screening using publicly available data. No capital commitment implied.

**Example**: Using GEOX to rank prospectivity of sub-basins in the Malay Basin using published well completion reports and open-access 2D seismic grids. Query outputs basin-averaged porosity trends and thermal gradient estimates.

**Action**: Auto-proceed. No human sign-off required.

**SLA**: Results returned immediately.

**Who signs off**: No sign-off required. Provenance still logged to vault_ledger.

---

### MEDIUM — Prospect Ranking / Farm-In

**Definition**: Prospect-level assessment informing a farm-in or license bidding decision. Involves proprietary or limited-access data.

**Example**: GEOX evaluates three prospects on a PSC block for ranking prior to a farm-in offer. Output includes estimated P50 net pay and GRV for each prospect. A human geoscientist must review before the ranking is used in commercial negotiations.

**Action**: 888 HOLD issued. Human sign-off required.

**SLA**: Authorized reviewer (Geoscientist L1+) must respond within **48 hours**.

**Who signs off**: Geoscience Lead or equivalent.

---

### HIGH — Resource Estimation / Drill Decision

**Definition**: Prospect-level resource volume estimation directly supporting a drill/no-drill capital decision.

**Example**: GEOX estimates P10/P50/P90 in-place volumes for a candidate well location. Output feeds into the Basis of Decision (BOD) for a well worth >USD 10M. Requires Chief Geoscientist and HSE review before drill recommendation proceeds.

**Action**: 888 HOLD. `human_signoff_required = True`. `regulator_notify = True`.

**SLA**: Chief Geoscientist + HSE sign-off within **5 business days**.

**Who signs off**: Chief Geoscientist (L2) + HSE representative.

---

### CRITICAL — Regulatory Filing / Reserve Certification

**Definition**: Any GEOX output used in a regulatory filing, reserves certification, or public disclosure (SPE-PRMS compliant).

**Example**: GEOX provides supporting analysis for a 2C contingent resource certification submitted to a national regulator. GEOX output is advisory only — a Competent Person (CP) must independently validate and sign the submission.

**Action**: 888 HOLD. `legal_review = True`. GEOX output is labelled "ADVISORY ONLY — requires CP validation".

**SLA**: Competent Person review + legal clearance within **15 business days**.

**Who signs off**: Competent Person (certified) + Legal + Regulator submission.

**Important**: GEOX cannot substitute for a Competent Person. It provides structured, audited, provenance-tagged supporting analysis only.

## 3. Human Sign-Off Workflow

Step-by-step procedure when `human_signoff_required = True`:

```
Step 1: GEOX issues 888 HOLD
  └── arifos_telemetry.hold = "888 HOLD"
  └── vault_ledger sealed with HOLD status

Step 2: Audit package generated by GeoXReporter
  ├── Markdown report (tables, floor checklist, 888 HOLD banner)
  ├── JSON audit (vault_ledger format, machine-readable)
  └── Human brief (3-paragraph plain language)

Step 3: Authorized reviewer notified
  └── Notification includes: request_id, prospect_name, risk_level, verdict, SLA deadline

Step 4: Reviewer examines
  ├── GeoResponse (insights, predictions, confidence)
  ├── Provenance chain (source of every quantity)
  ├── Validator verdicts (why each insight is supported/ambiguous/contradicted)
  └── arifOS telemetry block

Step 5: Reviewer decision
  ├── APPROVE → vault_ledger sealed with APPROVED status
  │            downstream action permitted
  ├── REJECT  → vault_ledger sealed with REJECTED status
  │            request terminated, reason logged
  └── MORE DATA → GEOX re-runs with additional tools or constraints
                 new 888 HOLD if still medium/high/critical risk

Step 6: Decision sealed immutably
  └── vault_ledger entry: decision, reviewer_id, timestamp, justification
```

## 4. Regulator Escalation

When `risk_level = "critical"` and `regulator_notify = True`:

1. **Data sovereignty check**: Confirm all data sources are permitted for regulatory use (no data with export restrictions)
2. **Export control check**: If cross-border data involved, confirm no ITAR/EAR restrictions
3. **Legal hold**: All GEOX outputs for this request placed under legal hold — no deletion, no modification
4. **Competent Person package**: GeoXReporter generates a CP-ready package (provenance chain, methodology statement, uncertainty quantification, limitation disclaimer)
5. **Submission**: CP reviews GEOX package, adds independent validation, submits to regulator
6. **Post-submission**: GEOX audit trail retained for regulatory discovery period (typically 7+ years)

## 6. Data Sovereignty Rules

1. **Public-domain-first**: GEOX prefers openly licensed data (public geological surveys, published literature, open EO imagery). Proprietary data sources must be explicitly declared in `ProvenanceRecord.source_id`.

2. **No proprietary assumptions**: GEOX tools carry no embedded proprietary datasets. All data is injected at runtime via tool adapters.

3. **Cross-border data**: If the prospect location is in a jurisdiction with data export restrictions (e.g., upstream data under PSC confidentiality), the requester is responsible for compliance. GEOX flags cross-border queries in the audit trail.

4. **Provenance declaration**: Every `GeoQuantity` must have a `ProvenanceRecord`. Anonymous data (no declared source) is rejected by the validator (F2 Truth violation).

5. **Retention**: Vault_ledger entries are retained per the arifOS retention policy. GEOX does not purge audit records independently.

## 7. F13 Sovereign Veto

**F13 (Sovereign)** grants Arif (identity: `888_JUDGE`) the ultimate, unconditional veto over any GEOX action.

### How it works:
1. At any point in the pipeline, `888_JUDGE` can issue a veto command
2. Veto is transmitted via arifOS `apex_soul (888)` tool
3. GEOX receives the veto signal, halts all pending actions
4. Veto reason is logged immutably to vault_ledger
5. No downstream system may override or bypass a veto

### What a veto covers:
- Stopping a tool execution mid-run
- Blocking a GeoResponse from being returned to a client
- Cancelling a human sign-off that is in progress
- Retracting a previous SEAL verdict (if new evidence warrants)

### What a veto does NOT do:
- Delete vault_ledger entries (those are immutable per F1 Amanah)
- Retroactively change provenance records

## 8. Incident Response — VOID Verdict

When `GeoResponse.verdict = "VOID"`:

1. **Immediate**: 888 HOLD issued automatically. No output returned to client.
2. **Within 1 hour**: Senior geoscientist notified with full audit package.
3. **Within 24 hours**: Root cause analysis:
   - Which insight was contradicted?
   - Which tool returned the contradicting data?
   - Is the LLM synthesis making physically impossible claims?
   - Is the tool returning erroneous data?
4. **Corrective action plan** (within 48 hours):
   - If LLM synthesis error: adjust planner prompt / tool weighting
   - If tool error: validate tool data source, replace mock with better data
   - If genuine geological contradiction: document and present to requester as a geological uncertainty
5. **Vault_ledger**: VOID incident logged with root cause + corrective action

## 10. Anti-Hantu Clause (F9)

**F9 (Anti-Hantu)**: GEOX never claims consciousness, feelings, certainty, or soul. It is a symbolic reasoning instrument under human authority.

In practice:
- All `GeoInsight.text` outputs are tagged with epistemic status in `arifos_telemetry`: `CLAIM / PLAUSIBLE / HYPOTHESIS / ESTIMATE / UNKNOWN`
- GEOX never says "I know" — it says "the Earth Model indicates" or "the data suggests"
- Confidence values are always bounded: never 0.0 (overconfident rejection) or 1.0 (impossible certainty) in geological contexts
- The human brief generated by GeoXReporter always includes a limitation statement: "GEOX is a computational tool. These findings require geoscience review before any capital or regulatory decision."

|---------|-----|
| SEAL | CLAIM |
